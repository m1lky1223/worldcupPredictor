  <files>

  </files>
  <read_first>
    - scripts/seed.ts
    - packages/domain/src/db/index.ts
    - packages/domain/src/db/schema.ts
  </read_first>
  <action>
    Create a new TypeScript script at scripts/db-init.ts (per D-02, D-03, D-04). This script must:
    1. Resolve the database hostname dynamically: default to "postgres" if running inside a container, otherwise fall back to "localhost" (using a simple check on environment variables or files like IS_DOCKER).
    2. Execute database migrations programmatically using Drizzle's node-postgres migrator and the schemas from @worldcup/domain.
    3. Query the database to check if any teams exist. If the query returns 0 teams (seamless developer onboarding), read the seed datasets from "data/teams.json" and "data/fixtures.json" and perform inserts into the teams and matches tables. If teams already exist, log that seeding is skipped and terminate cleanly.
    Ensure errors are caught and logged, and the node process exits with code 0 on success or 1 on failure.
  </action>
  <acceptance_criteria>
    - The file scripts/db-init.ts must exist.
    - Running "tsx scripts/db-init.ts" against an empty database applies migrations and seeds teams and matches.
    - Running "tsx scripts/db-init.ts" against a populated database does not duplicate records or fail, and logs that seeding is skipped.
  </acceptance_criteria>
  <verify>
    npx tsx scripts/db-init.ts
  </verify>
  <done>
    Programmatic db-init script written, verified, and supports conditional seeding based on existing records.
  </done>
</task>

Ensure the service depends on "postgres" with "service_healthy" status. Add healthchecks to the "api", "web", and "mcp" services using a node-based TCP net.connect script (e.g. node -e "require('net').connect(PORT, 'localhost').on('error', () => process.exit(1)).on('connect', () => process.exit(0))" where PORT is 4000, 3000, and 4001 respectively) rather than wget to ensure reliable status reporting across all environments. Configure "api", "worker", "web", and "mcp" service configurations to depend on the completion of the "db-migrate" container (using "depends_on" with condition "service_completed_successfully"). Note that apps/mcp is a development service that shares the underlying codebase integrated under /api/mcp on port 3000 in apps/web.
  </action>

    - Services "api", "web", and "mcp" must define "healthcheck" configurations.
  </acceptance_criteria>
  <verify>
